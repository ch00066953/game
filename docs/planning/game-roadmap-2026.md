# 游戏开发路线图（2026）

面向目标：持续产出经典好玩游戏，并确保移动端与 PC 端都可玩、可维护、可扩展。

## 一、阶段目标

### 阶段 A：快速上新（2-4 周）
目标：用最短时间扩充可玩数量，沉淀统一脚手架与交互规范。

建议优先：
- 打砖块
- 2048
- 推箱子
- 泡泡龙（简化版）

交付标准：
- 每个新游戏都具备开始、暂停、重开、结算流程
- 同时支持键盘与触控
- 具备基础 README（玩法、控制、已知限制）

### 阶段 B：玩法深挖（4-8 周）
目标：提高单游戏耐玩度，增加成长感与复玩率。

建议方向：
- 关卡系统（难度曲线）
- 分数与排行榜（本地）
- 音效与反馈优化
- 教程与新手引导

建议游戏：
- 塔防基础版
- 平台跳跃进阶版
- 战棋小地图原型

交付标准：
- 每个重点游戏至少 3 个可区分难度档
- 关键行为有即时反馈（命中、受击、过关、失败）
- 移动端核心流程单手可完成

### 阶段 C：中型项目（8-16 周）
目标：进入系统复杂度更高的游戏类型，形成系列化能力。

建议方向：
- 卡牌 Roguelike 小体量版
- 轻量 RPG（剧情 + 存档）
- 生存建造简化版

交付标准：
- 有可持续迭代的数据结构与配置表
- 有明确版本节奏（v0.1、v0.2、v1.0）
- 有统一的缺陷回归清单

## 二、季度节奏建议

- 每周：1 个小版本（修复 + 体验优化）
- 每双周：1 个可见功能增量（关卡、角色、机制）
- 每月：1 次专题迭代（移动端体验、性能、UI 统一）

## 三、选题原则

1. 先做高反馈、低复杂度题材，快速形成正循环。
2. 新项目尽量复用已有输入、UI、状态管理模式。
3. 任何新玩法先做最小可玩版本，再补视觉表现。
4. 双端适配不是后置项，必须在首版即完成。

## 四、文档协作规范

- 游戏立项先写一页策划（目标、玩法循环、MVP 范围）
- 开发前写任务拆分（按 2-6 小时粒度）
- 每次改动涉及玩法时同步更新对应 README
